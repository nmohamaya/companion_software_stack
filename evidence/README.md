# evidence/ — generated assurance reports

Canonical store for every generated assurance report, locally and in CI
(Issue #804). Everything in this folder **except this README is
gitignored** — evidence is generated, never versioned; in CI each
category is uploaded as a workflow artifact (`evidence-*`, 14-day
retention) because runners are ephemeral.

## Layout

```text
evidence/
  <category>/<YYYY-MM-DD_HHMMSSZ>_<git-short-sha>/   one run
    manifest.txt                                     provenance stamp
    <report files...>
  <category>/latest -> newest run                    local convenience symlink
```

The timestamp is UTC (sortable); the git short SHA ties every report to
the exact tree it was generated from. `manifest.txt` records timestamp,
SHA, branch, dirty-flag, host, generator script, and (in CI) the GitHub
run/job — written by `deploy/lib_evidence.sh`.

## Categories

| Category | Contents | Generated by |
|---|---|---|
| `safety-audit/` | `safety_audit_report.md` — 28-rule safety-critical C++ audit | `deploy/safety_audit.sh` (local); `safety-audit` CI job (advisory) |
| `tests/` | `ctest_<variant>.log` — full ctest output per variant (`default`, `asan`, `tsan`, `ubsan`, `coverage`) | `deploy/run_ci_local.sh` (local); `build-and-test` CI matrix |
| `coverage/` | `coverage.info` (lcov), `summary.txt`, `html/` report | `deploy/run_ci_local.sh --job COV` (local); `coverage` CI job |
| `static-analysis/` | `cppcheck.log` | `cppcheck` CI job (advisory) |

Candidate future categories (see Issue #804): `scenarios/` (Gazebo
`run_report.txt` copies), `sanitizers/` (failure excerpts), `benchmarks/`
(baseline-comparator output), release SBOM snapshots.

## Where do the reports come from? (local vs CI)

Evidence is generated **when the corresponding job runs** — this folder
only contains what has actually executed on this machine:

- `bash deploy/safety_audit.sh` → `safety-audit/`
- `bash deploy/run_ci_local.sh` (BUILD/ASAN/TSAN/UBSAN/COV jobs) →
  `tests/` + `coverage/`; `--job CPPCHECK` → `static-analysis/`
- GitHub CI runs generate the same evidence on the runner and upload it
  as `evidence-*` workflow artifacts (runners are ephemeral).

**To pull a CI run's evidence down into this folder:**

```bash
bash deploy/fetch_ci_evidence.sh              # newest CI run on this branch
bash deploy/fetch_ci_evidence.sh <run-id>     # specific run
bash deploy/fetch_ci_evidence.sh --pr <N>     # newest CI run of a PR
```

which lands under `evidence/ci/<run-id>/evidence-<category>/…`, with each
run's CI-written `manifest.txt` intact.

## Adding a new category

```bash
source deploy/lib_evidence.sh
dir=$(evidence_run_dir my-category)
my_tool > "${dir}/report.txt"
```

Then document the category in the table above and, if CI-generated, add
an `actions/upload-artifact` step named `evidence-<category>`.

## Retention

Local: nothing is auto-pruned — delete old run dirs freely (`latest`
symlinks re-point on the next run). CI artifacts expire per the
`retention-days` on each upload step.
